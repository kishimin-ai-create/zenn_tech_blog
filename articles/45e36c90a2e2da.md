## 作ったもの

TDD（テスト駆動開発）の Todo アプリを GitHub Copilot に作らせました。

**アプリ概要**
開発タスクを Todo として登録し、TDD ワークフローで一つずつこなしていくためのツールです。

**技術スタック**

| レイヤー       | 技術                                          |
| -------------- | --------------------------------------------- |
| フロントエンド | React 19 / TypeScript / Vite / TailwindCSS v4 |
| 状態管理       | TanStack Query / Jotai                        |
| バックエンド   | Hono / Node.js / TypeScript                   |
| DB             | MySQL                                         |
| テスト         | Vitest / Playwright / Storybook               |

**リポジトリ**

https://github.com/kishimin-ai-create/TDD_todo_app

## なぜバイブコーディングしたか

GitHub Copilot にできるだけコーディングを任せたら、どこまで開発を進められるのか試したくて始めました。

## 工夫した点

GitHub Copilot にテスト駆動開発（TDD）を行わせました。
テスト → 実装 → リファクタリングの流れを維持するように指示しました。

プリンシプルとして、KISS のような短い言葉を rules に記載しました。
AIに与える情報量をできるだけ少なくしたかったことと、短いルールが推奨されていたためです。

## 良くなかった点（AI）

ハッピーパスですら、期待通りに動かないケースが多くありました。

また、1つの技術記事に内容を詰め込みすぎることが多くありました。

さらに、以下のような問題もありました。

- コメントがほとんど追加されない
- リントエラーや型エラーが残ったまま終了する
- `.gitignore` 配下のファイルまで push しようとする
- `hono-openapi` を使わず、自作の OpenAPI 定義を作成する

その結果、プロンプトで修正指示を出すことが多くなりました。

## 良くなかった点（自分）

設計書をほとんど作成せず、プロンプトでの指示だけで開発を進めていました。

認証やログイン画面を含め、画面設計書も用意していませんでした。

その結果、実装の方向性がブレたり、意図した実装にならないことがありました。

## 次やるなら

rules と skills の違いを理解したうえで、それぞれを適切に設計したいです。

また、以下も改善したいです。

- セキュリティ対策を考慮した実装をさせたい
- 小さい粒度でコミットさせたい
- 適切なコメントを書かせたい

## 学び

CIでのリントや型チェックは必須だと感じました。

また、AIはコード生成速度は高いですが、生成結果をそのまま信用すると、後からの修正やリファクタリングのコストが大きくなりました。

結果として、実装よりも修正に時間を使うことが多かったです。
